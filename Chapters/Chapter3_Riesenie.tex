\section{Architektúra systému}
\label{sec:Architektura}

Pred samotnou implementáciou som navrhol celkovú architektúru systému, ktorá definuje tok dát od GPS trackerov až po webovú aplikáciu. Pochopenie tejto architektúry je kľúčové pre orientáciu v~nasledujúcich sekciách, pretože jednotlivé komponenty na seba nadväzujú a~spolupracujú.

\subsection{Prehľad komponentov}
\label{sec:PrehladKomponent}

Systém som navrhol ako viacvrstvovú architektúru pozostávajúcu zo šiestich hlavných vrstiev. Každá vrstva má jasne definovanú zodpovednosť a~komunikuje s~ostatnými vrstvami cez štandardizované protokoly. Tento prístup zabezpečuje modularitu, jednoduchú údržbu a~možnosť nezávislého škálovania jednotlivých komponentov.

\begin{enumerate}
    \item \textbf{Koncové zariadenia (End Devices)} -- SenseCap T1000-B GPS trackery, ktoré zbierajú pozičné dáta pomocou GPS modulu a~odosielajú ich cez LoRaWAN protokol. Tieto zariadenia sú optimalizované pre nízku spotrebu energie a~dlhodobú prevádzku na batérii.

    \item \textbf{LoRaWAN Gateway} -- RAK7289 V2 slúži ako most medzi bezdrôtovými rádiovými signálmi a~IP sieťou. Gateway prijíma LoRaWAN pakety, dekóduje ich a~preposiela na MQTT broker. Zároveň obsahuje integrovaný Network Server pre správu pripojených zariadení.

    \item \textbf{MQTT Broker} -- Firemný broker na adrese mqtt.hedurio.com sprostredkúva asynchrónnu komunikáciu medzi gateway a~backend aplikáciou. Využíva model publish/subscribe, ktorý je ideálny pre IoT scenáre s~množstvom producentov dát.

    \item \textbf{Backend Server} -- Python aplikácia postavená na frameworku Flask, ktorá prijíma správy z~MQTT brokera, parsuje ich, validuje a~ukladá do databázy. Zároveň poskytuje REST API pre frontend a~WebSocket spojenie pre real-time aktualizácie.

    \item \textbf{Databáza} -- PostgreSQL 16 s~rozšíreniami TimescaleDB pre efektívne ukladanie časových rád a~PostGIS pre geospatial operácie. Táto kombinácia umožňuje efektívne spracovanie veľkého množstva GPS záznamov a~priestorové dotazy.

    \item \textbf{Frontend} -- Single Page Application postavená na React 19 s~mapovou knižnicou Leaflet.js, ktorá poskytuje interaktívne používateľské rozhranie pre vizualizáciu pozícií, histórie pohybu a~správu geofencing zón.
\end{enumerate}

\subsection{Dátový tok v~systéme}
\label{sec:DatovyTok}

Pre lepšie pochopenie fungovania systému je dôležité popísať životný cyklus GPS správy od jej vzniku až po zobrazenie na mape. Tento proces som vizualizoval pomocou sekvenčného diagramu, ktorý znázorňuje postupnosť krokov a~komunikáciu medzi jednotlivými komponentmi.

Životný cyklus GPS správy prechádza nasledujúcimi fázami:

\begin{enumerate}
    \item \textbf{Získanie GPS fixu} -- Tracker aktivuje GPS modul a~čaká na získanie pozície zo satelitov. Pri cold start môže tento proces trvať 30-120 sekúnd, pri warm start len niekoľko sekúnd.

    \item \textbf{Zostavenie payloadu} -- Po získaní fixu tracker zostaví dátový paket obsahujúci GPS súradnice, stav batérie, teplotu a~ďalšie telemetrické údaje v~proprietárnom formáte SenseCap.

    \item \textbf{LoRaWAN prenos} -- Payload je odoslaný cez LoRaWAN protokol na frekvenčnom pásme EU868. Tracker používa adaptívny spreading factor pre optimalizáciu dosahu a~spotreby energie.

    \item \textbf{Príjem gateway} -- Gateway prijme rádiovú správu, dekóduje LoRaWAN paket a~extrahuje užitočné dáta. Zároveň pridá metadáta o~kvalite signálu (RSSI, SNR).

    \item \textbf{MQTT publikácia} -- Gateway publikuje dekódovaný JSON na MQTT topic. Správa obsahuje identifikáciu zariadenia, dekódované dáta a~informácie o~prenose.

    \item \textbf{Spracovanie backendom} -- Backend ako MQTT subscriber prijme správu, spustí parser na extrakciu relevantných informácií a~vykoná validácie.

    \item \textbf{Uloženie do databázy} -- Validované dáta sú uložené do PostgreSQL. Trigger automaticky vytvorí PostGIS POINT z~GPS súradníc a~aktualizuje last\_seen zariadenia.

    \item \textbf{WebSocket broadcast} -- Backend cez Socket.IO odošle novú pozíciu všetkým pripojeným klientom. Zároveň skontroluje geofence pravidlá a~prípadne vygeneruje alerty.

    \item \textbf{Aktualizácia UI} -- Frontend prijme WebSocket správu a~aktualizuje marker na mape s~plynulou animáciou prechodu na novú pozíciu.
\end{enumerate}

Celý proces od získania GPS fixu po zobrazenie na mape trvá typicky menej ako 2~sekundy, čo je pre live tracking plne dostačujúce. Najväčšiu latenciu spôsobuje samotné získanie GPS fixu a~LoRaWAN prenos, ktoré sú však mimo kontroly softvérovej časti systému.

Prvou fázou projektu bola konfigurácia hardvérovej infraštruktúry -- LoRaWAN gateway a~GPS trackerov. \subsection{Teoretický základ: LoRaWAN protokol}
\label{sec:UvodLoRaWAN}

LoRaWAN (Long Range Wide Area Network) je bezdrôtový komunikačný protokol navrhnutý špeciálne pre IoT zariadenia s~nízkym energetickým odberom a~požiadavkou na dlhý dosah~\cite{lorawan_spec}. Protokol je štandardizovaný organizáciou LoRa Alliance a~v~súčasnosti patrí medzi najrozšírenejšie LPWAN (Low Power Wide Area Network) technológie na svete.

\subsubsection{Porovnanie s~inými IoT technológiami}

Pre výber vhodnej komunikačnej technológie som analyzoval niekoľko alternatív. Každá má svoje špecifické výhody a~nevýhody, ktoré predurčujú jej vhodnosť pre konkrétne použitie.

Pre projekt GPS tracking som zvolil LoRaWAN z~nasledujúcich dôvodov:

\begin{itemize}
    \item \textbf{Dlhý dosah} -- Na eventoch s~rozlohou niekoľko kilometrov je nevyhnutné pokryť celú plochu jednou gateway. LoRaWAN dosahuje v~Line of Sight podmienkach až 15~km, v~mestskom prostredí 2-5~km, čo je pre väčšinu eventov dostačujúce.

    \item \textbf{Nízka spotreba energie} -- Trackery musia vydržať na batérii minimálne jeden pracovný deň (8-12 hodín) bez možnosti dobíjania. LoRaWAN Class A zariadenia väčšinu času spia a~prebúdzajú sa len na odoslanie dát, čo dramaticky znižuje spotrebu.

    \item \textbf{Nelicencované pásmo} -- Pásmo EU868 je v~Európe voľne dostupné pre IoT aplikácie bez potreby licencie, čo znižuje prevádzkové náklady a~administratívnu záťaž.

    \item \textbf{Škálovateľnosť} -- Jedna gateway dokáže obslúžiť tisíce zariadení, čo je dôležité pre veľké eventy s~množstvom trackovaných osôb.
\end{itemize}

WiFi a~Bluetooth som zamietol kvôli nedostatočnému dosahu -- pokrytie rozľahlého areálu by vyžadovalo desiatky access pointov. Sigfox som zamietol kvôli obmedzenej obojsmernej komunikácii a~závislosti na externom operátorovi.

\subsubsection{Architektúra LoRaWAN}

LoRaWAN architektúra pozostáva zo štyroch vrstiev, ktoré spolupracujú na prenose dát od koncového zariadenia po aplikačný server:

\begin{enumerate}
    \item \textbf{End Device} -- GPS tracker alebo senzor zbierajúci dáta. Komunikuje s~gateway pomocou rádiového signálu na frekvencii 868~MHz (v~Európe).

    \item \textbf{Gateway} -- Prijíma rádiové signály a~konvertuje ich na IP pakety. Gateway nie je inteligentná -- iba preposiela dáta na Network Server.

    \item \textbf{Network Server} -- Spravuje sieť, deduplikuje správy (ak zariadenie počuje viacero gateways), overuje autentifikáciu a~routuje dáta na správny Application Server.

    \item \textbf{Application Server} -- Spracováva aplikačné dáta. V~mojom prípade je to backend aplikácia, ktorá prijíma správy cez MQTT.
\end{enumerate}

    \item \textbf{Packet Forwarder} -- Nastavenie parametrov pre optimálny príjem LoRaWAN paketov vrátane spreading faktorov a~bandwidth.
\end{itemize}

\subsubsection{OTAA registrácia}

Registráciu trackerov som realizoval pomocou OTAA (Over-The-Air Activation) autentifikácie. OTAA je preferovaná metóda oproti ABP (Activation By Personalization), pretože session kľúče sú generované dynamicky pri každom pripojení zariadenia k~sieti. To zvyšuje bezpečnosť -- ak by útočník zachytil komunikáciu, kľúče sa pri nasledujúcom pripojení zmenia.

OTAA vyžaduje tri parametre:

\begin{itemize}
    \item \textbf{DevEUI} -- Unikátny 64-bitový identifikátor zariadenia priradený výrobcom. Je vytlačený na štítku trackera.

    \item \textbf{AppEUI (JoinEUI)} -- Identifikátor aplikácie alebo join servera. Definuje, ku ktorej aplikácii sa zariadenie pripája.

    \item \textbf{AppKey} -- 128-bitový šifrovací kľúč pre odvodenie session kľúčov. Musí byť rovnaký na zariadení aj na Network Serveri.
\end{itemize}

Proces registrácie som realizoval v~týchto krokoch:

\begin{enumerate}
    \item Vytvorenie aplikácie na gateway Network Server s~definovaným AppEUI
    \item Pridanie trackera ako End Device s~DevEUI z~nálepky na zariadení
    \item Konfigurácia trackera cez mobilnú aplikáciu SenseCraft (Bluetooth)
    \item Nastavenie rovnakého AppEUI a~AppKey v~trackeri
    \item Aktivácia trackera a~čakanie na Join Request/Accept handshake
\end{enumerate}

Pri úspešnom pripojení tracker prehrá krátku melódiu signalizujúcu úspešnú registráciu do siete.

\subsection{Problémy a~ich riešenia}
\label{sec:HWProblemy}

Počas konfigurácie hardvéru som narazil na niekoľko problémov, ktoré vyžadovali hlbšie štúdium dokumentácií a~konzultácie. V~tejto sekcii popisujem najvýznamnejšie z~nich a~spôsoby, akými som ich vyriešil.

\subsubsection{GPS Cold Start}

Pri prvom zapnutí alebo po dlhšej neaktivite tracker posielal súradnice (0.0, 0.0), kým GPS modul nezískal fix. Toto je štandardné správanie pri cold start -- GPS modul nemá uložené údaje o~polohe satelitov (almanach a~efemeridy) a~potrebuje čas na ich stiahnutie.

Cold start môže trvať 30-120 sekúnd v~závislosti od viditeľnosti oblohy. Problém som vyriešil implementáciou filtrovania nulových súradníc v~parseri -- pozície (0.0, 0.0) sú ignorované a~neukladajú sa do databázy.

\subsubsection{Firmware Bug -- Battery Stuck}

Podľa diskusií na komunitnom fóre SenseCraft niektoré trackery hlásili 100\% batérie aj keď boli takmer vybité. Tento firmware bug spôsoboval nepríjemné prekvapenia, keď sa trackery náhle vypli napriek indikovanej plnej batérii.

Implementoval som detekciu tejto anomálie v~backend aplikácii -- ak tracker hlási 100\% batériu viac ako 24 hodín bez akejkoľvek zmeny, systém ho označí ako podozrivý a~upozorní operátora.

\subsubsection{GPS Presnosť v~Budovách}

Presnosť GPS vnútri budov dosahovala len približne 300 metrov, čo výrazne komplikuje tracking v~interiéroch. Toto je fyzikálne obmedzenie GPS technológie -- signál zo satelitov je príliš slabý na preniknutie cez steny a~stropy budov.

Pre testovanie počas vývoja som pravidelne chodil von, kde presnosť bola konzistentne pod 10 metrov. Pre produkčné nasadenie na outdoor eventoch tento problém nie je relevantný. Pre prípadné indoor použitie by bolo potrebné implementovať WiFi alebo Bluetooth lokalizáciu, ktorú tracker T1000-B podporuje, ale vyžaduje databázu známych access pointov.

\section{MQTT a~Parser}
\label{sec:MQTTParser}

Po úspešnej konfigurácii hardvéru som pristúpil k~implementácii softvérovej časti pre príjem a~spracovanie dát. Táto fáza zahŕňala vytvorenie MQTT klienta pre príjem správ z~gateway, vývoj robustného parsera pre extrakciu relevantných dát a~implementáciu validácií pre ošetrenie rôznych edge cases.

\subsection{Teoretický základ: MQTT Protokol}
\label{sec:MQTTProtokol}

MQTT (Message Queuing Telemetry Transport) je ľahký komunikačný protokol navrhnutý pre IoT aplikácie a~prostredia s~obmedzenými zdrojmi~\cite{mqtt_spec}. Protokol vyvinula firma IBM v~roku 1999 a~od roku 2014 je štandardizovaný organizáciou OASIS.

\subsubsection{Publish/Subscribe Model}

MQTT využíva model publish/subscribe, ktorý sa zásadne líši od tradičného request/response modelu používaného v~REST API:

\begin{itemize}
    \item \textbf{REST API (polling)} -- Klient musí periodicky dopytovať server na nové dáta. To spôsobuje zbytočnú záťaž a~latenciu.

    \item \textbf{MQTT (push)} -- Klient sa prihlási na odber topicu a~broker mu automaticky doručí správy hneď ako sú publikované. Niet potreby dopytovať sa.
\end{itemize}

V~MQTT existujú tri typy účastníkov:

\begin{enumerate}
    \item \textbf{Publisher} -- Zariadenie alebo aplikácia, ktorá publikuje správy na topic. V~mojom systéme je publisherom gateway.

    \item \textbf{Subscriber} -- Zariadenie alebo aplikácia, ktorá odoberá správy z~topicu. V~mojom systéme je subscriberom backend aplikácia.

    \item \textbf{Broker} -- Centrálny server, ktorý prijíma správy od publisherov a~distribuuje ich subscriberom. Používam firemný broker mqtt.hedurio.com.
\end{enumerate}

\subsection{Implementácia MQTT Klienta}
\label{sec:BrokerPripojenie}

Pre pripojenie na firemný MQTT broker som použil Python knižnicu \texttt{paho-mqtt}, ktorá je de facto štandardom pre MQTT v~Python ekosystéme. Knižnica podporuje všetky verzie protokolu MQTT (3.1, 3.1.1, 5.0) a~poskytuje asynchronné callback rozhranie.

Pripojenie na broker vyžadovalo konfiguráciu TLS šifrovania s~klientskými certifikátmi. Broker vyžaduje vzájomnú autentifikáciu -- nielen server sa preukazuje certifikátom klientovi, ale aj klient serveru. Toto zvyšuje bezpečnosť a~zabraňuje neoprávnenému prístupu k~dátam.

Po úspešnom pripojení sa klient prihlási na odber topicu \texttt{application/SENSECAP/decoded}, kde gateway publikuje dekódované správy z~trackerov. Každá správa je vo formáte JSON a~obsahuje identifikáciu zariadenia, dekódované dáta a~metadáta o~prenose.

Nasledujúca ukážka kódu demonštruje inicializáciu MQTT klienta s~TLS zabezpečením:

\begin{lstlisting}[language=Python, label=src:mqtt-client, caption={Inicializácia MQTT klienta s~TLS autentifikáciou}]
import paho.mqtt.client as mqtt
import ssl

class MQTTClient:
    def __init__(self, config):
        self.client = mqtt.Client(protocol=mqtt.MQTTv311)

        # TLS konfigurácia s klientskými certifikátmi
        self.client.tls_set(
            ca_certs=config.CA_CERT,      # Certifikát CA
            certfile=config.CLIENT_CERT,   # Klientský certifikát
            keyfile=config.CLIENT_KEY,     # Klientský kľúč
            tls_version=ssl.PROTOCOL_TLSv1_2
        )

        # Callback funkcie
        self.client.on_connect = self._on_connect
        self.client.on_message = self._on_message

    def _on_connect(self, client, userdata, flags, rc):
        if rc == 0:
            # Prihlásenie na odber GPS dát
            client.subscribe("application/SENSECAP/decoded")

    def _on_message(self, client, userdata, msg):
        # Parsovanie a spracovanie payloadu
        self.parser.process(msg.payload)
\end{lstlisting}

\subsection{Spracovanie Dekódovaných Dát}
\label{sec:Parser}

Spracovanie dát z~trackerov prebieha v~dvoch fázach prostredníctvom dvojice parserov. Prvý parser je nasadený na firemnom MQTT brokeri mqtt.hedurio.com a~vychádza z~open-source riešenia SenseCap decoder od Seeed Studio (GitHub repozitár Seeed-Solution/SenseCAP-Decoder), ktoré bolo upravené pre naše potreby. Tento parser transformuje raw LoRaWAN payload na štruktúrovaný JSON formát -- preto topic nesie označenie \texttt{decoded}.

Druhý parser, ktorého implementáciu popisujem v~tejto sekcii, prijíma už dekódované JSON správy a~extrahuje z~nich relevantné dáta pre GPS tracking. Štruktúra JSON obsahuje pole \texttt{messages} s~unikátnymi identifikátormi typu merania (measurement ID), z~ktorých extrahujeme GPS súradnice a~telemetriu.

Parser iteruje cez pole \texttt{messages}, extrahuje hodnoty podľa measurement ID a~zostavuje štruktúrovaný objekt s~GPS pozíciou a~telemetriou. Ak niektoré povinné pole chýba (latitude alebo longitude), parser vráti \texttt{None} a~správa je zahodená.

\begin{lstlisting}[language=Python, label=src:parser, caption={Extrakcia GPS súradníc z~SenseCap payloadu}]
MEASUREMENT_IDS = {
    'LATITUDE': '4198',
    'LONGITUDE': '4197',
    'BATTERY': '3000'
}

def extract_gps(measurements: list) -> dict:
    """Extrahuje GPS a telemetriu z measurements array"""
    latitude = longitude = battery = None

    for m in measurements:
        mid = m.get('measurementId')
        val = m.get('measurementValue')

        if mid == MEASUREMENT_IDS['LATITUDE']:
            latitude = val
        elif mid == MEASUREMENT_IDS['LONGITUDE']:
            longitude = val
        elif mid == MEASUREMENT_IDS['BATTERY']:
            battery = val

    # Validácia povinných polí
    if latitude is None or longitude is None:
        return None

    # Validácia rozsahu (WGS84)
    if not (-90 <= latitude <= 90 and -180 <= longitude <= 180):
        return None

    return {'lat': latitude, 'lon': longitude, 'battery': battery}
\end{lstlisting}

\subsubsection{Detekcia GPS Driftu}

Špeciálnu pozornosť som venoval detekcii GPS driftu -- situácie, keď tracker hlási pozíciu vzdialenú stovky metrov od skutočnej polohy. Toto sa deje pri slabom GPS signáli, napríklad v~blízkosti vysokých budov alebo v~hustom lese.

Implementoval som detekciu založenú na výpočte rýchlosti medzi dvoma po sebe idúcimi pozíciami. Ak by zariadenie muselo cestovať rýchlosťou vyššou ako 500 km/h, aby sa dostalo z~predchádzajúcej pozície na aktuálnu, považujem novú pozíciu za neplatnú a~zahodím ju.

Tento prístup má svoje limitácie -- nezachytí pomalý drift, kde sa pozícia posúva postupne. Pre komplexnejšie riešenie by bolo potrebné implementovať Kalman filter alebo podobný algoritmus, čo som ponechal ako možné budúce rozšírenie.

\subsection{TimescaleDB -- Časové Rady}
\label{sec:TimescaleDB}

TimescaleDB je open-source rozšírenie PostgreSQL optimalizované pre časové rady (time-series data). GPS pozície sú klasickým príkladom časových rád -- každý záznam má timestamp a~nové záznamy neustále pribúdajú, pričom staršie dáta sa väčšinou len čítajú.

\subsubsection{Koncept Hypertables}

Kľúčovým konceptom TimescaleDB sú hypertables -- špeciálne tabuľky, ktoré sa automaticky rozdeľujú na menšie chunks podľa času. Externe vyzerajú ako bežné PostgreSQL tabuľky, ale interne sú optimalizované pre časové dotazy.

Výhody hypertables:

\begin{itemize}
    \item \textbf{Automatické partitioning} -- Dáta sa automaticky rozdeľujú do chunks podľa času bez manuálneho zásahu.

    \item \textbf{Efektívne dotazy} -- Query na posledných 24 hodín prehľadá len relevantné chunks namiesto celej tabuľky.

    \item \textbf{Kompresia} -- Staršie chunks môžu byť komprimované, čo ušetrí až 95\% miesta.

    \item \textbf{Retention policies} -- Automatické mazanie starých dát podľa definovaných pravidiel.
\end{itemize}

Pre tabuľku \texttt{positions} som nastavil chunk interval 1 týždeň, kompresiu pre dáta staršie ako 7 dní a~retention policy pre dáta staršie ako 90 dní.

\begin{lstlisting}[language=SQL, label=src:hypertable, caption={Vytvorenie TimescaleDB hypertable s~kompresiou}]
-- Konverzia bežnej tabuľky na hypertable
SELECT create_hypertable(
    'positions',
    'timestamp',
    chunk_time_interval => INTERVAL '1 week',
    if_not_exists => TRUE
);

-- Aktivácia automatickej kompresie
ALTER TABLE positions SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_id'
);

-- Politika kompresie: dáta staršie ako 7 dní
SELECT add_compression_policy('positions', INTERVAL '7 days');

-- Politika retencie: mazanie dát starších ako 90 dní
SELECT add_retention_policy('positions', INTERVAL '90 days');
\end{lstlisting}

\subsection{PostGIS -- Geospatial Operácie}
\label{sec:PostGIS}

PostGIS je rozšírenie PostgreSQL pre prácu s~geografickými dátami. Namiesto ukladania GPS súradníc do samostatných stĺpcov ako FLOAT, PostGIS poskytuje natívny dátový typ \texttt{GEOGRAPHY}, ktorý umožňuje efektívne priestorové dotazy.

\subsubsection{Prečo nie FLOAT stĺpce?}

Na prvý pohľad by sa mohlo zdať jednoduchšie uložiť latitude a~longitude do dvoch FLOAT stĺpcov. Tento prístup má však zásadné nevýhody:

\begin{itemize}
    \item \textbf{Výpočet vzdialenosti} -- Vzdialenosť na guľovom povrchu Zeme nie je triviálna -- vyžaduje Haversine vzorec alebo Vincenty formulu. PostGIS toto rieši funkciou \texttt{ST\_Distance}.

    \item \textbf{Radius search} -- Nájsť zariadenia v~okruhu 5~km od bodu vyžaduje komplexný výpočet. PostGIS poskytuje \texttt{ST\_DWithin}.

    \item \textbf{Point-in-polygon} -- Detekcia či bod leží v~polygóne (geofencing) je výpočtovo náročná. PostGIS má \texttt{ST\_Contains}.

    \item \textbf{Indexy} -- GIST indexy na GEOGRAPHY stĺpcoch sú rádovo efektívnejšie ako B-tree indexy na FLOAT stĺpcoch.
\end{itemize}

\subsubsection{Triggery a~Automatizácia}

Pre zníženie komplexity aplikačnej logiky som implementoval databázové triggery, ktoré automaticky vykonávajú určité operácie:

\begin{itemize}
    \item \textbf{populate\_location} -- Pri INSERT do \texttt{positions} automaticky vytvorí PostGIS POINT z~lat/lon súradníc.

    \item \textbf{update\_last\_seen} -- Pri INSERT do \texttt{positions} aktualizuje \texttt{last\_seen} timestamp v~tabuľke \texttt{devices}.
\end{itemize}

Tento prístup zabezpečuje konzistenciu dát aj v~prípade, že by sa niekto pokúsil vložiť dáta priamo do databázy bez použitia aplikácie.

\begin{lstlisting}[language=SQL, label=src:trigger, caption={PostGIS trigger pre automatické vytvorenie geografického bodu}]
-- Funkcia: vytvorí PostGIS POINT z lat/lon
CREATE OR REPLACE FUNCTION populate_location()
RETURNS TRIGGER AS $$
BEGIN
    NEW.location = ST_SetSRID(
        ST_MakePoint(NEW.longitude, NEW.latitude),
        4326  -- WGS84 súradnicový systém
    )::geography;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Trigger: spustí sa pred každým INSERT
CREATE TRIGGER set_location
BEFORE INSERT OR UPDATE ON positions
FOR EACH ROW EXECUTE FUNCTION populate_location();
\end{lstlisting}

Pre funkcionalitu ``Nearby search'' (vyhľadanie zariadení v~okruhu od bodu) som vytvoril databázovú funkciu využívajúcu PostGIS operátor \texttt{ST\_DWithin}:

\begin{lstlisting}[language=SQL, label=src:nearby, caption={PostGIS funkcia pre vyhľadanie zariadení v~okruhu}]
CREATE OR REPLACE FUNCTION get_nearby_devices(
    target_lat DOUBLE PRECISION,
    target_lon DOUBLE PRECISION,
    max_distance_km REAL DEFAULT 5.0
) RETURNS TABLE (
    device_id TEXT, device_name TEXT,
    latitude DOUBLE PRECISION, longitude DOUBLE PRECISION,
    distance_km REAL
) AS $$
BEGIN
    RETURN QUERY
    SELECT d.device_id, d.device_name, p.latitude, p.longitude,
           (ST_Distance(p.location,
               ST_SetSRID(ST_MakePoint(target_lon, target_lat), 4326)
           ) / 1000.0)::REAL AS distance_km
    FROM latest_positions p
    JOIN devices d ON p.device_id = d.device_id
    WHERE ST_DWithin(p.location,
        ST_SetSRID(ST_MakePoint(target_lon, target_lat), 4326)::geography,
        max_distance_km * 1000);  -- konverzia km na metre
END;
$$ LANGUAGE plpgsql;
\end{lstlisting}

\subsection{WebSocket Real-time}
\label{sec:WebSocket}

Pre real-time aktualizácie pozícií som integroval Flask-SocketIO~\cite{socketio_docs}. WebSocket protokol umožňuje obojsmernú komunikáciu medzi serverom a~klientom bez nutnosti opakovaného nadväzovania spojenia.

\subsubsection{Prečo WebSocket a~nie Polling}

Tradičný polling prístup, kde klient periodicky dopytuje server na nové dáta, má niekoľko nevýhod:

\begin{itemize}
    \item \textbf{Latencia} -- Ak interval pollingu je 5 sekúnd, priemerná latencia je 2.5 sekundy. Pre live tracking je to príliš veľa.

    \item \textbf{Záťaž servera} -- Pri 100 klientoch a~1-sekundovom intervale server obsluhuje 100 requestov za sekundu, aj keď nie sú žiadne nové dáta.

    \item \textbf{Spotreba batérie} -- Na mobilných zariadeniach neustále HTTP requesty výrazne zvyšujú spotrebu.
\end{itemize}

WebSocket tieto problémy rieši -- spojenie je persistentné a~server odosiela dáta len keď sú dostupné.

\subsubsection{WebSocket Events}

Implementoval som nasledujúce WebSocket eventy:

\begin{itemize}
    \item \textbf{new\_position} -- Odoslaný všetkým klientom pri prijatí novej GPS pozície
    \item \textbf{geofence\_alert} -- Odoslaný pri vstupe/výstupe zariadenia zo zóny
    \item \textbf{device\_status} -- Odoslaný pri zmene stavu zariadenia (online/offline)
    \item \textbf{initial\_state} -- Odoslaný novému klientovi pri pripojení s~aktuálnym stavom
\end{itemize}

Nasledujúca ukážka demonštruje broadcast novej pozície všetkým pripojeným klientom:

\begin{lstlisting}[language=Python, label=src:websocket, caption={WebSocket broadcast novej GPS pozície}]
from flask_socketio import SocketIO, emit

socketio = SocketIO(app, cors_allowed_origins="*")

def broadcast_position(position_data: dict):
    """Odošle novú pozíciu všetkým pripojeným klientom"""
    socketio.emit('new_position', {
        'device_id': position_data['device_id'],
        'latitude': position_data['latitude'],
        'longitude': position_data['longitude'],
        'battery': position_data['battery'],
        'timestamp': position_data['timestamp'].isoformat()
    }, namespace='/')

def broadcast_geofence_alert(alert_data: dict):
    """Odošle geofence alert pri vstupe/výstupe zo zóny"""
    socketio.emit('geofence_alert', {
        'device_id': alert_data['device_id'],
        'zone_name': alert_data['zone_name'],
        'event_type': alert_data['event_type'],  # 'entry'/'exit'
        'timestamp': alert_data['timestamp'].isoformat()
    }, namespace='/')
\end{lstlisting}

\subsection{Connection Pooling}
\label{sec:ConnectionPooling}

Pre efektívnu prácu s~databázou som použil connection pool z~knižnice psycopg3. Connection pooling udržiava vopred vytvorené databázové spojenia, čo eliminuje overhead vytvárania nového spojenia pre každý request.

Nakonfiguroval som pool s~minimálne 2 a~maximálne 10 spojeniami. Pri štarte aplikácie sa vytvorí 2 spojenia, ďalšie sa vytvárajú podľa potreby až do maxima. Nepoužívané spojenia sa po timeout uvoľňujú.

\section{Geofencing a~POI}
\label{sec:GeofencingPOI}

Geofencing je jednou z~najhodnotnejších funkcionalít systému pre praktické nasadenie na eventoch. Umožňuje definovať virtuálne zóny a~automaticky detekovať pohyb zariadení.

\subsection{Koncept Geofencing}
\label{sec:GeofencingKoncept}

Geofencing je technika vytvárania virtuálnych geografických hraníc (geofences) a~automatickej detekcie, keď zariadenie do definovanej oblasti vstúpi alebo ju opustí. Názov vznikol spojením slov ``geographic'' a~``fencing'' (oplotenie).

Pre eventy má geofencing široké praktické využitie:

\begin{itemize}
    \item \textbf{Bezpečnostné zóny} -- Detekcia vstupu do backstage area, VIP zón alebo zakázaných priestorov
    \item \textbf{Monitoring personálu} -- Sledovanie, či security personál pokrýva pridelené sektory
    \item \textbf{Štatistiky návštevnosti} -- Počítanie návštevníkov v~jednotlivých zónach (stage, bar, food court)
    \item \textbf{Perimeter alert} -- Upozornenie ak trackované osoby opustia definovaný areál
\end{itemize}

\subsection{Implementácia Geofencing}
\label{sec:GeofencingImpl}

Pre kreslenie zón som implementoval intuitívne rozhranie inšpirované štandardnými kresliacimi aplikáciami.

Proces vytvorenia zóny:

    \item Používateľ klikne na tlačidlo ``Draw New Zone'' v~paneli nástrojov
    \item Kurzor sa zmení na krížik indikujúci režim kreslenia
    \item Kliknutím na mapu sa pridávajú body polygónu
    \item Dvojklikom alebo kliknutím na prvý bod sa polygón uzavrie
    \item Zobrazí sa formulár pre zadanie názvu, farby a~popisu zóny
    \item Po potvrdení sa zóna uloží do databázy ako PostGIS polygon
\end{enumerate}

\subsubsection{Entry/Exit Detekcia}

Pri každej novej GPS pozícii backend kontroluje, či zariadenie vstúpilo alebo opustilo niektorú z~definovaných zón. Detekcia využíva PostGIS funkciu \texttt{ST\_Contains}, ktorá efektívne testuje, či bod leží vnútri polygónu.

Algoritmus detekcie:

\begin{enumerate}
    \item Získanie aktuálneho stavu -- v~ktorých zónach sa zariadenie nachádza
    \item Kontrola novej pozície voči všetkým zónam pomocou \texttt{ST\_Contains}
    \item Porovnanie nového a~predchádzajúceho stavu
    \item Pre nové zóny (entry) -- vytvorenie ENTRY eventu
    \item Pre opustené zóny (exit) -- vytvorenie EXIT eventu
    \item Aktualizácia aktuálneho stavu v~databáze
    \item WebSocket broadcast alertu všetkým pripojeným klientom
\end{enumerate}

Nasledujúca ukážka demonštruje SQL dotaz pre detekciu zón obsahujúcich danú pozíciu:

\begin{lstlisting}[language=SQL, label=src:geofence-detect, caption={PostGIS detekcia zariadenia v~geofence zónach}]
-- Nájdi všetky zóny obsahujúce danú GPS pozíciu
SELECT g.id, g.name, g.zone_type
FROM geofences g
WHERE ST_Contains(
    g.polygon,
    ST_SetSRID(ST_MakePoint($1, $2), 4326)::geography
) AND g.active = TRUE;

-- Parametre: $1 = longitude, $2 = latitude
-- Výsledok: zoznam zón, v ktorých sa zariadenie nachádza
\end{lstlisting}

\subsection{Docker Compose Setup}

\begin{itemize}
    \item \textbf{Reprodukovateľnosť} -- Prostredie je identické na lokálnom stroji, testovacom serveri aj v~produkcii
    \item \textbf{Izolácia} -- Každá služba beží v~samostatnom kontajneri s~definovanými závislosťami
    \item \textbf{Jednoduchá distribúcia} -- Nový vývojár spustí prostredie jedným príkazom \texttt{docker-compose up}
    \item \textbf{Škálovateľnosť} -- Jednotlivé služby možno škálovať nezávisle
\end{itemize}

Konzultant odporučil tento prístup ako firemný štandard pre nové projekty.

\subsection{Bezpečnostné Opatrenia}
\label{sec:Bezpecnost}

Implementoval som niekoľko bezpečnostných opatrení:

\begin{itemize}
    \item \textbf{TLS šifrovanie} -- MQTT komunikácia na porte 8883 je šifrovaná s~klientskými certifikátmi
    \item \textbf{Parameterized queries} -- Všetky SQL dotazy používajú parametrizované statements ako ochranu pred SQL injection
    \item \textbf{CORS} -- Backend akceptuje requesty len z~definovaných origins
    \item \textbf{Environment variables} -- Citlivé údaje (heslá, API kľúče) sú v~.env súbore mimo version control
    \item \textbf{Input validation} -- Všetky vstupy z~API sú validované na backende
\end{itemize}

\section{Testovanie}
\label{sec:Testovani}

Testovanie prebiehalo kontinuálne počas celého vývoja. Okrem manuálneho testovania som vytvoril GPS simulátor pre automatizované testy a~realizoval real-world testy s~fyzickými trackermi.

\subsection{GPS Simulátor}
\label{sec:Simulator}

Pre vývoj a~testovanie bez fyzických trackerov som vytvoril GPS simulátor v~Pythone. Simulátor generuje realistické testovacie dáta vrátane:

\begin{itemize}
    \item Náhodný pohyb po definovanom území s~nastaviteľnou rýchlosťou
    \item Simulácia GPS driftu pre testovanie validácií
    \item Generovanie edge cases (cold start, battery drain)
    \item Možnosť definovania trasy pomocou waypointov
\end{itemize}

Simulátor výrazne urýchlil vývoj, pretože som nemusel čakať na reálne payloady z~trackerov a~mohol som testovať scenáre, ktoré sa v~reálnom prostredí vyskytujú zriedka.

\subsection{Real-world Testing}
\label{sec:RealTesting}

Okrem simulátora som systém testoval s~reálnymi trackermi v~teréne. Tieto testy odhalili niekoľko dôležitých poznatkov:

\begin{itemize}
    \item GPS presnosť v~budovách je výrazne horšia -- až 300 metrov oproti <10 metrov outdoor
    \item Batéria vydrží približne 2-3 dni pri 5-minútovom reporting intervale
    \item LoRaWAN dosah v~mestskom prostredí je cca 2-3 km, v~Line of Sight až 10 km
    \item Cold start GPS trvá 30-120 sekúnd v~závislosti od viditeľnosti oblohy
    \item Multipath efekt v~blízkosti vysokých budov spôsobuje občasný GPS drift
\end{itemize}

Tieto poznatky som zohľadnil v~implementácii validácií a~odporúčaniach pre produkčné nasadenie.
