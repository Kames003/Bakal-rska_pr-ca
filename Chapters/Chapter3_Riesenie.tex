\section{Hardware Setup a~Konfigurácia}
\label{sec:HardwareSetup}

\subsection{Úvod do LoRaWAN}
\label{sec:UvodLoRaWAN}

LoRaWAN (Long Range Wide Area Network) je bezdrôtový komunikačný protokol navrhnutý pre IoT zariadenia s~nízkym energetickým odberom~\cite{lorawan_spec}. Na rozdiel od WiFi či Bluetooth, ktoré majú dosah desiatky metrov, LoRaWAN umožňuje komunikáciu na vzdialenosti až 15 kilometrov v~otvorenom teréne.

Architektúra LoRaWAN pozostáva zo štyroch vrstiev:
\begin{itemize}
    \item \textbf{End Device} -- GPS tracker zbierajúci pozičné dáta
    \item \textbf{Gateway} -- most medzi rádiovými signálmi a~IP sieťou
    \item \textbf{Network Server} -- správa siete a~routovanie správ
    \item \textbf{Application Server} -- spracovanie dát aplikáciou
\end{itemize}

Pre tento projekt som zvolil LoRaWAN kvôli jeho dlhému dosahu a~nízkej spotrebe energie trackerov, čo je kritické pre celodenné eventy kde nie je možné priebežné nabíjanie zariadení.

\subsection{Gateway RAK7289 V2}
\label{sec:Gateway}

Firma mi poskytla gateway RAK7289 V2 vo variante C2 s~integrovaným LTE modulom pre pripojenie do internetu. Prvým krokom bola aktualizácia firmware z~verzie 2.2.2 na aktuálnu 2.2.14, ktorá prinášala opravy bezpečnostných chýb a~lepšiu stabilitu MQTT integrácie.

Konfigurácia gateway zahŕňala:
\begin{itemize}
    \item Nastavenie frekvenčného plánu \textbf{EU868} pre európsky región
    \item Konfiguráciu MQTT Bridge pre odosielanie dát na broker \texttt{mqtt.hedurio.com}

\subsection{SenseCap T1000-B Trackery}
\label{sec:Trackery}

Pre tracking osôb som používal trackery SenseCap T1000-B od firmy Seeed Studio~\cite{sensecap_datasheet}. Tieto kompaktné zariadenia kombinujú GPS, WiFi a~Bluetooth lokalizáciu s~LoRaWAN komunikáciou.

Registrácia trackerov prebiehala pomocou \textbf{OTAA} (Over-The-Air Activation) autentifikácie, ktorá vyžaduje tri parametre:
\begin{itemize}
    \item \textbf{DevEUI} -- unikátny identifikátor zariadenia
    \item \textbf{AppEUI} -- identifikátor aplikácie
    \item \textbf{AppKey} -- šifrovací kľúč pre komunikáciu
\end{itemize}

Konfiguráciu som realizoval cez mobilnú aplikáciu SenseCraft, ktorá komunikuje s~trackerom pomocou Bluetooth. Nastavil som uplink interval na 5 minút pre vývojové účely a~GPS scan timeout na 120 sekúnd pre zvýšenie šance získania presnej polohy.

\subsection{Problémy a~riešenia}
\label{sec:HWProblemy}

Počas konfigurácie hardvéru som narazil na niekoľko problémov:

\begin{enumerate}
    \item \textbf{GPS cold start} -- pri prvom zapnutí tracker posielal súradnice (0.0, 0.0), kým GPS modul nezískal fix. Toto som vyriešil filtráciou v~parseri.

    \item \textbf{Firmware bug batérie} -- podľa diskusií na SenseCraft fóre niektoré trackery hlásili 100\% batérie aj keď boli takmer vybité. Implementoval som detekciu tejto anomálie.

    \item \textbf{GPS v~budovách} -- presnosť GPS vnútri budov dosahovala len cca 300 metrov. Pre testovanie som preto chodil von, kde presnosť bola pod 10 metrov.
\end{enumerate}

\section{MQTT a~Parser}
\label{sec:MQTTParser}

\subsection{MQTT Protokol}
\label{sec:MQTTProtokol}

MQTT (Message Queuing Telemetry Transport) je ľahký publish/subscribe protokol ideálny pre IoT aplikácie~\cite{mqtt_spec}. Na rozdiel od REST API, kde klient aktívne dopytuje server, MQTT umožňuje serveru posielať dáta klientom okamžite po ich prijatí.

Pre pripojenie na firemný broker \texttt{mqtt.hedurio.com} som použil Python knižnicu \texttt{paho-mqtt} s~TLS šifrovaním:

\begin{lstlisting}[language=Python, caption={Inicializácia MQTT klienta s~TLS}]
import paho.mqtt.client as mqtt

client = mqtt.Client()
client.tls_set(ca_certs=CA_CERT_PATH)
client.username_pw_set(username, password)
client.on_connect = on_connect
client.on_message = on_message
client.connect("mqtt.hedurio.com", 8883, keepalive=60)
\end{lstlisting}

Po úspešnom pripojení sa klient prihlási na odber topicu \texttt{application/SENSECAP/decoded}, kde gateway publikuje dekódované správy z~trackerov.

\subsection{Parser SenseCap Payloadov}
\label{sec:Parser}

Správy z~trackerov prichádzajú vo formáte JSON s~vnorenou štruktúrou. GPS súradnice sú uložené v~poli \texttt{messages} s~unikátnymi identifikátormi:
\begin{itemize}
    \item \texttt{4198} = LATITUDE
    \item \texttt{4197} = LONGITUDE
    \item \texttt{3000} = BATTERY
\end{itemize}

\begin{lstlisting}[language=Python, caption={Extrakcia GPS súradníc z~payloadu}]
def _extract_gps_coordinates(self, measurements):
    latitude = longitude = battery = None

    for m in measurements:
        mid = m.get('measurementId')
        value = m.get('measurementValue')

        if mid == '4198':
            latitude = value
        elif mid == '4197':
            longitude = value
        elif mid == '3000':
            battery = value

    if latitude and longitude:
        return {'latitude': latitude,
                'longitude': longitude,
                'battery': battery}
    return None
\end{lstlisting}

\subsection{Validácie a~Edge Cases}
\label{sec:Validace}

Počas vývoja som identifikoval niekoľko edge cases, ktoré bolo potrebné ošetriť:

\begin{table}[htbp]
    \centering
    \caption{Edge cases a~ich riešenia v~parseri}
    \label{tab:edgecases}
    \begin{tabular}{|l|l|l|}
        \hline
        \textbf{Edge Case} & \textbf{Príčina} & \textbf{Riešenie} \\
        \hline
        GPS (0.0, 0.0) & Cold start & Filtrovanie nulových súradníc \\
        Battery 100\% stuck & Firmware bug & Detekcia >24h rovnakej hodnoty \\
        GPS drift & Slabý signál & Detekcia nereálnej rýchlosti \\
        Corrupted JSON & Prenosové chyby & Try/except s~logovaním \\
        Duplicitné správy & MQTT retry & Upsert namiesto INSERT \\
        \hline
    \end{tabular}
\end{table}

\begin{lstlisting}[language=Python, caption={Validácia GPS súradníc}]
def validate_coordinates(lat: float, lon: float) -> bool:
    """Validuje GPS suradnice podla WGS84"""
    if lat == 0.0 and lon == 0.0:
        return False  # Cold start
    return -90 <= lat <= 90 and -180 <= lon <= 180
\end{lstlisting}

Mentor ocenil tento prístup slovami: \emph{``Cool, to jsou věci na které jsem ani nemyslel.''} Robustný parser bol kľúčový pre spoľahlivosť celého systému.

\subsection{TimescaleDB}
\label{sec:TimescaleDB}

TimescaleDB~\cite{timescaledb_docs} je rozšírenie PostgreSQL optimalizované pre časové rady (time-series data). GPS pozície sú klasickým príkladom -- každý záznam má timestamp a~postupne pribúdajú nové dáta.

Hlavnou výhodou je koncept \textbf{hypertables}, ktoré automaticky rozdeľujú dáta do menších chunks podľa času:

\begin{lstlisting}[language=SQL, caption={Vytvorenie TimescaleDB hypertable}]
SELECT create_hypertable(
    'positions',
    'timestamp',
    chunk_time_interval => INTERVAL '1 week',
    if_not_exists => TRUE
);

-- Automaticka kompresia dat starsich ako 7 dni
ALTER TABLE positions SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_id'
);
SELECT add_compression_policy('positions', INTERVAL '7 days');
\end{lstlisting}

Mentor potvrdil správnosť voľby: \emph{``PostGIS a~TimescaleDB v~praktických záležitostech používáme, takže super volba.''}

\subsection{PostGIS}
\label{sec:PostGIS}

PostGIS~\cite{postgis_docs} rozširuje PostgreSQL o~geospatial funkcie. Namiesto ukladania súradníc do samostatných stĺpcov \texttt{latitude} a~\texttt{longitude} ako FLOAT, PostGIS umožňuje použiť typ \texttt{GEOGRAPHY(POINT, 4326)}, ktorý podporuje priestorové operácie.

Kľúčové funkcie, ktoré som využil:
\begin{itemize}
    \item \texttt{ST\_DWithin()} -- nájdenie zariadení v~okruhu od bodu
    \item \texttt{ST\_Contains()} -- detekcia či je bod vnútri polygónu (geofence)
    \item \texttt{ST\_Distance()} -- výpočet vzdialenosti medzi bodmi
\end{itemize}

\begin{lstlisting}[language=SQL, caption={PostGIS nearby search -- zariadenia do 5km}]
SELECT device_id, device_name,
    ST_Distance(
        location,
        ST_SetSRID(ST_MakePoint(18.16, 49.82), 4326)::geography
    ) / 1000.0 AS distance_km
FROM latest_positions
WHERE ST_DWithin(
    location,
    ST_SetSRID(ST_MakePoint(18.16, 49.82), 4326)::geography,
    5000  -- 5km v metroch
)
ORDER BY distance_km;
\end{lstlisting}

def broadcast_position(position_data):

\subsection{Geofencing koncept}
\label{sec:GeofencingKoncept}

Geofencing je technika vytvárania virtuálnych geografických zón a~automatickej detekcie, keď zariadenie do zóny vstúpi alebo ju opustí. Pre eventy je to užitočné napríklad pre:
\begin{itemize}
    \item Detekciu vstupu do backstage area
    \item Monitorovanie pohybu security personálu
    \item Štatistiky návštevnosti jednotlivých zón
\end{itemize}

\subsection{Implementácia}
\label{sec:GeofencingImplementace}

Zóny sú uložené ako PostGIS polygóny. Pri každej novej pozícii backend kontroluje, či zariadenie vstúpilo alebo opustilo niektorú zónu:

\begin{lstlisting}[language=SQL, caption={Detekcia zariadení v~geofence zóne}]
SELECT d.device_id, d.device_name
FROM devices d
JOIN latest_positions lp ON d.device_id = lp.device_id
JOIN geofences g ON ST_Contains(g.polygon, lp.location::geometry)
WHERE g.id = %s;
\end{lstlisting}

\subsection{GPS Simulátor}
\label{sec:Simulator}

Pre vývoj bez fyzických trackerov som vytvoril GPS simulátor, ktorý generuje realistické testovanie dáta:

\begin{lstlisting}[language=Python, caption={Generovanie simulovanej GPS pozície}]
def generate_position(base_lat, base_lon, radius_m=100):
    """Generuje nahodnu poziciu v okruhu"""
    angle = random.uniform(0, 2 * math.pi)
    distance = random.uniform(0, radius_m)

    delta_lat = (distance / 111000) * math.cos(angle)
    delta_lon = (distance / 111000) * math.sin(angle)

    return {
        'latitude': base_lat + delta_lat,
        'longitude': base_lon + delta_lon,
        'battery': random.randint(20, 100)
    }
\end{lstlisting}
