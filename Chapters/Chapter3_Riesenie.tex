\section{Architektúra systému}
\label{sec:Architektura}

Pred samotnou implementáciou som navrhol celkovú architektúru systému, ktorá definuje tok dát od GPS trackerov až po webovú aplikáciu. Pochopenie tejto architektúry je kľúčové pre orientáciu v~nasledujúcich sekciách.

\subsection{Prehľad komponentov}
\label{sec:PrehladKomponent}

Systém pozostáva zo šiestich hlavných vrstiev, ktoré spolupracujú na zbere, spracovaní a~vizualizácii GPS dát:

\begin{enumerate}
    \item \textbf{End Devices (Trackery)} -- SenseCap T1000-B GPS trackery zbierajúce pozičné dáta
    \item \textbf{LoRaWAN Gateway} -- RAK7289 V2 most medzi rádiovými signálmi a~IP sieťou
    \item \textbf{MQTT Broker} -- mqtt.hedurio.com sprostredkúva správy medzi gateway a~backendom
    \item \textbf{Backend Server} -- Python/Flask aplikácia spracujúca dáta a~poskytujúca API
    \item \textbf{Databáza} -- PostgreSQL s~rozšíreniami TimescaleDB a~PostGIS
    \item \textbf{Frontend} -- React webová aplikácia pre vizualizáciu a~správu
\end{enumerate}

\subsection{Dátový tok}
\label{sec:DatovyTok}

Životný cyklus GPS správy prechádza nasledujúcimi fázami:

\begin{enumerate}
    \item Tracker získa GPS fix a~zostaví payload s~pozíciou, batériou a~metadátami
    \item Payload je odoslaný cez LoRaWAN na frekvenčnom pásme EU868
    \item Gateway prijme rádiovú správu a~preloží ju do IP paketu
    \item Gateway publikuje dekódovaný JSON na MQTT topic
    \item Backend (MQTT subscriber) prijme správu a~spustí parser
    \item Parser validuje dáta a~extrahuje relevantné informácie
    \item Dáta sú uložené do PostgreSQL databázy
    \item WebSocket broadcastuje novú pozíciu všetkým pripojeným klientom
    \item Frontend aktualizuje mapu a~UI v~reálnom čase
\end{enumerate}

Táto architektúra zabezpečuje latenciu pod 2~sekundy od získania GPS fixu po zobrazenie na mape, čo je pre live tracking kritické.

\section{Hardware Setup a~Konfigurácia}

Prvou fázou projektu bola konfigurácia hardvérovej infraštruktúry -- LoRaWAN gateway a~GPS trackerov. Táto fáza vyžadovala pochopenie LoRaWAN protokolu, štúdium dokumentácií a~riešenie niekoľkých neočakávaných problémov.

\subsection{Úvod do LoRaWAN}
\label{sec:UvodLoRaWAN}

LoRaWAN (Long Range Wide Area Network) je bezdrôtový komunikačný protokol navrhnutý pre IoT zariadenia s~nízkym energetickým odberom~\cite{lorawan_spec}. Na rozdiel od WiFi či Bluetooth, ktoré majú dosah desiatky metrov, LoRaWAN umožňuje komunikáciu na vzdialenosti až 15~kilometrov v~otvorenom teréne.

Kľúčové charakteristiky LoRaWAN:
\begin{itemize}
    \item \textbf{Dlhý dosah} -- až 15~km v~Line of Sight podmienkach, 2-5~km v~mestskom prostredí
    \item \textbf{Nízka spotreba} -- trackery vydržia na batérii dni až týždne
    \item \textbf{Nízka dátová priepustnosť} -- vhodné pre malé payloady (GPS súradnice, telemetria)
    \item \textbf{Nelicencované pásmo} -- EU868 v~Európe nevyžaduje licenciu
\end{itemize}

Pre tento projekt som zvolil LoRaWAN kvôli jeho dlhému dosahu a~nízkej spotrebe energie trackerov. Na celodenných eventoch nie je možné priebežné nabíjanie zariadení, takže energetická efektívnosť bola prioritou.

Architektúra LoRaWAN pozostáva zo štyroch vrstiev:
\begin{itemize}
    \item \textbf{End Device} -- GPS tracker zbierajúci pozičné dáta
    \item \textbf{Gateway} -- most medzi rádiovými signálmi a~IP sieťou
    \item \textbf{Network Server} -- správa siete a~routovanie správ
    \item \textbf{Application Server} -- spracovanie dát aplikáciou
\end{itemize}

\subsubsection{OTAA registrácia}

Registrácia trackerov prebiehala pomocou \textbf{OTAA} (Over-The-Air Activation) autentifikácie. OTAA je bezpečnejšia alternatíva k~ABP (Activation By Personalization), pretože session kľúče sú generované dynamicky pri každom pripojení.

OTAA vyžaduje tri parametre:
\begin{itemize}
    \item \textbf{DevEUI} -- unikátny 64-bitový identifikátor zariadenia (výrobcom priradený)
    \item \textbf{AppEUI} (JoinEUI) -- identifikátor aplikácie/join servera
    \item \textbf{AppKey} -- 128-bitový šifrovací kľúč pre komunikáciu
\end{itemize}

Konfiguráciu trackerov som realizoval cez mobilnú aplikáciu SenseCraft, ktorá komunikuje s~trackerom pomocou Bluetooth. Proces registrácie:

\begin{enumerate}
    \item Vytvorenie aplikácie na gateway Network Server s~definovaným AppEUI
    \item Pridanie trackera ako End Device s~DevEUI z~nálepky na zariadení
    \item Nastavenie rovnakého AppEUI v~trackeri cez SenseCraft app
    \item Čakanie na Join Request/Accept handshake
\end{enumerate}

Pri úspešnom pripojení tracker prehrá charakteristickú melódiu. Dokumentácia to popisuje ako \emph{``A quick and cheerful melody when the network is successfully joined''} -- či je to ``cheerful'' je diskutabilné, ale aspoň signalizuje úspech.

\subsubsection{Konfigurácia parametrov}

Pre vývojové účely som nastavil nasledujúce parametre:

\begin{itemize}
    \item \textbf{Work Mode} -- Periodic (pravidelné odosielanie)
    \item \textbf{Uplink Interval} -- 5 minút (pre vývoj, produkčne 10-15 min)
    \item \textbf{GPS Scan Timeout} -- 120 sekúnd (zvýšené z~default 60s)
    \item \textbf{Heartbeat Interval} -- 60 minút
\end{itemize}

Zvýšenie GPS Scan Timeout na 120 sekúnd výrazne zlepšilo úspešnosť získania GPS fixu, hlavne pri cold start situáciách.

\subsection{Problémy a~riešenia}
\label{sec:HWProblemy}

\subsubsection{GPS cold start}

Pri prvom zapnutí tracker posielal súradnice (0.0, 0.0), kým GPS modul nezískal fix. Toto je bežné správanie pri ``cold start'' -- GPS modul nemá uložené údaje o~satelitoch a~potrebuje čas na ich vyhľadanie.

Riešenie implementované v~parseri:
\begin{lstlisting}[language=Python, caption={Filtrovanie cold start súradníc}]
def validate_coordinates(lat: float, lon: float) -> bool:
    """Validuje GPS suradnice - filtruje cold start a garbage values"""
    if lat == 0.0 and lon == 0.0:
        return False  # Cold start - tracker este nema GPS fix
    if lat == 999.0 or lon == 999.0:
        return False  # Garbage values
    return -90 <= lat <= 90 and -180 <= lon <= 180
\end{lstlisting}

Podľa diskusií na SenseCraft fóre niektoré trackery hlásili 100\% batérie aj keď boli takmer vybité. Tento firmware bug spôsobil na eventoch nepríjemné prekvapenia, keď sa trackery náhle vypli.

Implementoval som detekciu tejto anomálie -- ak tracker hlási 100\% batériu viac ako 24 hodín bez zmeny, systém ho označí ako podozrivý.

\section{MQTT a~Parser}
\label{sec:MQTTParser}

Po úspešnej konfigurácii hardvéru som pristúpil k~implementácii softvérovej časti pre príjem a~spracovanie dát. Táto fáza zahŕňala pripojenie na MQTT broker, vývoj robustného parsera a~ošetrenie množstva edge cases.

\subsection{MQTT Protokol}
\label{sec:MQTTProtokol}

MQTT (Message Queuing Telemetry Transport) je ľahký publish/subscribe protokol ideálny pre IoT aplikácie~\cite{mqtt_spec}. Na rozdiel od REST API, kde klient aktívne dopytuje server (polling), MQTT umožňuje serveru posielať dáta klientom okamžite po ich prijatí (push).

Výhody MQTT pre tento projekt:
\begin{itemize}
    \item \textbf{Nízka latencia} -- správy doručené v~milisekundách
    \item \textbf{Efektivita} -- minimálna hlavička protokolu (2 bajty)
    \item \textbf{Spoľahlivosť} -- QoS úrovne garantujúce doručenie
    \item \textbf{TLS podpora} -- šifrovaná komunikácia
\end{itemize}

\subsection{Pripojenie na broker}
\label{sec:BrokerPripojenie}

Pre pripojenie na firemný broker \texttt{mqtt.hedurio.com} som použil Python knižnicu \texttt{paho-mqtt} s~TLS šifrovaním:

\begin{lstlisting}[language=Python, caption={Kompletná inicializácia MQTT klienta s~TLS a~callback funkciami}]
import paho.mqtt.client as mqtt
import ssl

class MQTTClient:
    def __init__(self, config):
        self.client = mqtt.Client(client_id=config.CLIENT_ID)

        # TLS konfigurácia s certifikátmi
        self.client.tls_set(
            ca_certs=config.CA_CERT_PATH,
            certfile=config.CLIENT_CERT_PATH,
            keyfile=config.CLIENT_KEY_PATH,
            tls_version=ssl.PROTOCOL_TLS
        )

        # Autentifikácia
        self.client.username_pw_set(
            config.MQTT_USERNAME,
            config.MQTT_PASSWORD
        )

        # Callback funkcie
        self.client.on_connect = self._on_connect
        self.client.on_message = self._on_message
        self.client.on_disconnect = self._on_disconnect

    def _on_connect(self, client, userdata, flags, rc):
        if rc == 0:
            logger.info("Connected to MQTT broker")
            # Subscribe na topic s dekódovanými správami
            client.subscribe("application/SENSECAP/decoded", qos=1)
        else:
            logger.error(f"Connection failed: {rc}")

    def connect(self):
        self.client.connect(
            host="mqtt.hedurio.com",
            port=8883,
            keepalive=60
        )
        self.client.loop_start()
\end{lstlisting}

\begin{lstlisting}[language=Python, caption={Extrakcia GPS súradníc a~telemetrie z~SenseCap payloadu}]
class SenseCapParser:
    MEASUREMENT_IDS = {
        '4198': 'latitude',
        '4197': 'longitude',
        '3000': 'battery',
        '4175': 'temperature',
        '4190': 'light'
    }

    def parse_payload(self, payload: dict) -> Optional[dict]:
        """Parsuje surový payload z MQTT a extrahuje GPS + telemetriu"""
        try:
            device_eui = payload.get('devEUI')
            device_name = payload.get('deviceName', 'Unknown')
            measurements = payload.get('messages', [])

            # Extrakcia hodnôt podľa measurement ID
            data = {}
            for m in measurements:
                mid = str(m.get('measurementId'))
                value = m.get('measurementValue')

                if mid in self.MEASUREMENT_IDS:
                    data[self.MEASUREMENT_IDS[mid]] = value

            # Validácia povinných polí
            if not self._validate_gps(data):
                return None

            return {
                'device_eui': device_eui,
                'device_name': device_name,
                'latitude': data['latitude'],
                'longitude': data['longitude'],
                'battery': data.get('battery'),
                'temperature': data.get('temperature'),
                'timestamp': datetime.utcnow()
            }

        except Exception as e:
            logger.error(f"Parse error: {e}")
            self.stats['parse_errors'] += 1
            return None
\end{lstlisting}

\begin{lstlisting}[language=Python, caption={Komplexná validácia GPS súradníc vrátane detekcie driftu}]
def validate_gps_data(self, data: dict) -> bool:
    """Komplexná validácia GPS dát vrátane edge cases"""
    lat = data.get('latitude')
    lon = data.get('longitude')

    # Základná validácia prítomnosti
    if lat is None or lon is None:
        return False

    # Cold start filtrovanie
    if lat == 0.0 and lon == 0.0:
        logger.debug("Filtered cold start coordinates")
        return False

    # Garbage values
    if abs(lat) > 90 or abs(lon) > 180:
        logger.warning(f"Invalid coordinates: {lat}, {lon}")
        return False

    # Detekcia nereálnej rýchlosti (>500 km/h)
    if self._detect_gps_drift(data):
        logger.warning("GPS drift detected, filtering")
        return False

    return True

def _detect_gps_drift(self, data: dict) -> bool:
    """Detekuje GPS drift porovnaním s predchádzajúcou pozíciou"""
    device_eui = data.get('device_eui')
    if device_eui not in self.last_positions:
        return False

    last = self.last_positions[device_eui]
    distance = self._haversine_distance(
        last['lat'], last['lon'],
        data['latitude'], data['longitude']
    )
    time_diff = (data['timestamp'] - last['timestamp']).seconds

    if time_diff > 0:
        speed_kmh = (distance / time_diff) * 3600
        if speed_kmh > 500:  # Nereálna rýchlosť
            return True

    return False
\end{lstlisting}

class ParserStatistics:

\subsection{Výber technológií}

\subsubsection{Prečo TimescaleDB}

TimescaleDB~\cite{timescaledb_docs} je rozšírenie PostgreSQL optimalizované pre časové rady (time-series data). GPS pozície sú klasickým príkladom -- každý záznam má timestamp a~postupne pribúdajú nové dáta.

Výhody TimescaleDB:
\begin{itemize}
    \item \textbf{Hypertables} -- automatické rozdelenie dát do chunks podľa času
    \item \textbf{Kompresia} -- až 95\% úspora miesta pre staršie dáta
    \item \textbf{Continuous aggregates} -- predpočítané agregácie
    \item \textbf{Retention policies} -- automatické mazanie starých dát
\end{itemize}

\subsubsection{Prečo PostGIS}

PostGIS~\cite{postgis_docs} rozširuje PostgreSQL o~geospatial funkcie. Namiesto ukladania súradníc do samostatných stĺpcov ako FLOAT, PostGIS umožňuje použiť natívny dátový typ \texttt{GEOGRAPHY}.

\subsection{TimescaleDB Hypertables}
\label{sec:Hypertables}

Tabuľka \texttt{positions} je konvertovaná na TimescaleDB hypertable s~týždňovými chunks:

\begin{lstlisting}[language=SQL, caption={Vytvorenie TimescaleDB hypertable pre GPS pozície}]
-- Vytvorenie základnej tabuľky
CREATE TABLE positions (
    id BIGSERIAL,
    device_id INTEGER NOT NULL REFERENCES devices(id),
    latitude DOUBLE PRECISION NOT NULL,
    longitude DOUBLE PRECISION NOT NULL,
    location GEOGRAPHY(POINT, 4326),
    battery INTEGER,
    temperature REAL,
    rssi INTEGER,
    snr REAL,
    timestamp TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (id, timestamp)
);

-- Konverzia na hypertable
SELECT create_hypertable(
    'positions',
    'timestamp',
    chunk_time_interval => INTERVAL '1 week',
    if_not_exists => TRUE
);

-- Automatická kompresia dát starších ako 7 dní
ALTER TABLE positions SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_id',
    timescaledb.compress_orderby = 'timestamp DESC'
);

SELECT add_compression_policy('positions', INTERVAL '7 days');

-- Retention policy - mazanie dát starších ako 90 dní
SELECT add_retention_policy('positions', INTERVAL '90 days');
\end{lstlisting}

Výhody hypertables:
\begin{itemize}
    \item Query na posledných 24 hodín prehľadá len 1 chunk namiesto celej tabuľky
    \item Kompresia ušetrí až 90\% miesta pre historické dáta
    \item Automatické mazanie starých dát bez manuálneho zásahu
\end{itemize}

\subsection{PostGIS Geospatial Funkcie}
\label{sec:PostGISFunkcie}

PostGIS poskytuje množstvo funkcií pre prácu s~geografickými dátami:

\begin{lstlisting}[language=SQL, caption={PostGIS trigger pre automatické vytvorenie GEOGRAPHY z~lat/lon}]
-- Trigger pre automatické vytvorenie PostGIS POINT
CREATE OR REPLACE FUNCTION populate_location()
RETURNS TRIGGER AS $$
BEGIN
    NEW.location := ST_SetSRID(
        ST_MakePoint(NEW.longitude, NEW.latitude),
        4326
    )::geography;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_populate_location
    BEFORE INSERT OR UPDATE ON positions
    FOR EACH ROW
    EXECUTE FUNCTION populate_location();
\end{lstlisting}

\begin{lstlisting}[language=SQL, caption={PostGIS nearby search -- nájdenie zariadení v~okruhu}]
-- Nájdenie všetkých zariadení do 5 km od bodu
CREATE OR REPLACE FUNCTION get_nearby_devices(
    center_lat DOUBLE PRECISION,
    center_lon DOUBLE PRECISION,
    radius_meters INTEGER DEFAULT 5000
)
RETURNS TABLE (
    device_id INTEGER,
    device_name VARCHAR,
    distance_meters DOUBLE PRECISION,
    last_seen TIMESTAMPTZ
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        d.id,
        d.name,
        ST_Distance(
            lp.location,
            ST_SetSRID(ST_MakePoint(center_lon, center_lat), 4326)::geography
        ) AS distance,
        lp.timestamp
    FROM devices d
    JOIN LATERAL (
        SELECT location, timestamp
        FROM positions p
        WHERE p.device_id = d.id
        ORDER BY timestamp DESC
        LIMIT 1
    ) lp ON true
    WHERE ST_DWithin(
        lp.location,
        ST_SetSRID(ST_MakePoint(center_lon, center_lat), 4326)::geography,
        radius_meters
    )
    ORDER BY distance;
END;
$$ LANGUAGE plpgsql;
\end{lstlisting}

\subsection{Connection Pooling}
\label{sec:ConnectionPooling}

Pre efektívnu prácu s~databázou som použil connection pool z~knižnice \texttt{psycopg3}:

\begin{lstlisting}[language=Python, caption={Implementácia connection poolu s~dict row factory}]
from psycopg_pool import ConnectionPool
from psycopg.rows import dict_row

class DatabaseConnection:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._init_pool()
        return cls._instance

    def _init_pool(self):
        """Inicializácia connection poolu"""
        self.pool = ConnectionPool(
            conninfo=os.getenv('DATABASE_URL'),
            min_size=2,      # Minimum 2 spojenia vždy ready
            max_size=10,     # Maximum 10 paralelných spojení
            timeout=30,      # Timeout pre získanie spojenia
            kwargs={
                'row_factory': dict_row  # Výsledky ako dict
            }
        )

    def execute(self, query: str, params: tuple = None):
        """Vykonanie query s automatickým connection managementom"""
        with self.pool.connection() as conn:
            with conn.cursor() as cur:
                cur.execute(query, params)
                if cur.description:
                    return cur.fetchall()
                conn.commit()
                return cur.rowcount
\end{lstlisting}

from flask_socketio import SocketIO, emit, join_room

socketio = SocketIO()

@socketio.on('connect')
def handle_connect():
    """Handler pre nové WebSocket spojenie"""
    logger.info(f"Client connected: {request.sid}")
    # Odoslanie aktuálneho stavu všetkých trackerov
    trackers = db.get_all_trackers_with_last_position()
    emit('initial_state', {'trackers': trackers})

@socketio.on('subscribe_tracker')
def handle_subscribe(data):
    """Prihlásenie na odber konkrétneho trackera"""
    tracker_id = data.get('tracker_id')
    join_room(f'tracker_{tracker_id}')

def broadcast_new_position(position_data: dict):
    """Broadcast novej pozície všetkým pripojeným klientom"""
    socketio.emit('new_position', {
        'device_id': position_data['device_id'],

        'latitude': position_data['latitude'],
        'longitude': position_data['longitude'],
        'battery': position_data['battery'],
        'timestamp': position_data['timestamp'].isoformat()
    })

    # Notifikácia pre geofence eventy
    if position_data.get('geofence_event'):
        socketio.emit('geofence_alert', {
            'type': position_data['geofence_event']['type'],
            'zone_name': position_data['geofence_event']['zone_name'],
            'device_name': position_data['device_name']
        })

\section{Geofencing a~POI}
\label{sec:GeofencingPOI}

Geofencing je jednou z~najhodnotnejších funkcionalít systému pre praktické nasadenie na eventoch. Umožňuje definovať virtuálne zóny a~automaticky detekovať pohyb zariadení.

\subsection{Koncept Geofencing}
\label{sec:GeofencingKoncept}

Geofencing je technika vytvárania virtuálnych geografických zón a~automatickej detekcie, keď zariadenie do zóny vstúpi alebo ju opustí. Pre eventy má široké využitie:

\begin{itemize}
    \item Detekcia vstupu do backstage area (len authorized personál)
    \item Monitorovanie pohybu security personálu
    \item Štatistiky návštevnosti jednotlivých zón (stage, bar, toalety)
    \item Alerting pri opustení definovaného perimetra
\end{itemize}

\subsection{Entry/Exit Detekcia}
\label{sec:EntryExitDetekcia}

Pri každej novej pozícii backend kontroluje, či zariadenie vstúpilo alebo opustilo niektorú zónu:

\begin{lstlisting}[language=Python, caption={Detekcia geofence eventov pomocou PostGIS}]
def check_geofence_events(device_id: int, new_position: dict):
    """Kontrola entry/exit pre všetky zóny"""

    # Získanie aktuálneho stavu
    current_zones = db.get_device_current_zones(device_id)

    # Kontrola či je nová pozícia v nejakej zóne
    new_zones = db.execute("""
        SELECT g.id, g.name
        FROM geofences g
        WHERE ST_Contains(
            g.polygon::geometry,
            ST_SetSRID(ST_MakePoint(%s, %s), 4326)
        )
    """, (new_position['longitude'], new_position['latitude']))

    new_zone_ids = {z['id'] for z in new_zones}
    current_zone_ids = {z['id'] for z in current_zones}

    # Detekcia ENTRY (nové zóny)
    for zone_id in new_zone_ids - current_zone_ids:
        zone = next(z for z in new_zones if z['id'] == zone_id)
        create_geofence_event(device_id, zone_id, 'ENTRY')
        broadcast_geofence_alert('ENTRY', zone['name'], device_id)

    # Detekcia EXIT (opustené zóny)
    for zone_id in current_zone_ids - new_zone_ids:
        zone = next(z for z in current_zones if z['id'] == zone_id)
        create_geofence_event(device_id, zone_id, 'EXIT')
        broadcast_geofence_alert('EXIT', zone['name'], device_id)

    # Update aktuálneho stavu
    db.update_device_zones(device_id, list(new_zone_ids))
\end{lstlisting}

\section{Testovanie}

\subsection{GPS Simulátor}
\label{sec:Simulator}

Pre vývoj bez fyzických trackerov som vytvoril GPS simulátor:

\begin{lstlisting}[language=Python, caption={GPS simulátor generujúci realistické testovacie dáta}]
import random
import math
import time

class GPSSimulator:
    def __init__(self, base_lat: float, base_lon: float):
        self.current_lat = base_lat
        self.current_lon = base_lon

    def generate_next_position(self, speed_kmh: float = 5):
        """Generuje ďalšiu pozíciu simulujúcu pohyb"""
        # Náhodný smer pohybu
        angle = random.uniform(0, 2 * math.pi)

        # Vzdialenosť za 1 sekundu pri danej rýchlosti
        distance_m = (speed_kmh * 1000) / 3600

        # Prepočet na stupne
        delta_lat = (distance_m / 111000) * math.cos(angle)
        delta_lon = (distance_m / (111000 * math.cos(
            math.radians(self.current_lat)))) * math.sin(angle)

        self.current_lat += delta_lat
        self.current_lon += delta_lon

        return {
            'latitude': self.current_lat,
            'longitude': self.current_lon,
            'battery': random.randint(20, 100),
            'timestamp': datetime.utcnow()
        }

    def simulate_route(self, waypoints: list, interval_sec: int = 5):
        """Simuluje pohyb po definovanej trase"""
        for waypoint in waypoints:
            while self._distance_to(waypoint) > 10:
                self._move_towards(waypoint)
                yield self.get_current_position()
                time.sleep(interval_sec)
\end{lstlisting}

\subsection{Real-world Testing}
\label{sec:RealTesting}

Okrem simulátora som systém testoval s~reálnymi trackermi v~teréne. Testy odhalili:

\begin{itemize}
    \item Presnosť GPS v~budovách je výrazne horšia (až 300m)
    \item Batéria vydrží približne 2-3 dni pri 5-minútovom intervale
    \item LoRaWAN dosah je v~mestskom prostredí cca 2-3 km
    \item Cold start GPS trvá 30-120 sekúnd
\end{itemize}

Mentor potvrdil reálne parametre dosahu: \emph{``To můžu potvrdit, mám 10 km v~LoS.''}
