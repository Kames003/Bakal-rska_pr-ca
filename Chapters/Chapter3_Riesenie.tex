\section{Architektúra systému}
\label{sec:Architektura}

Pred samotnou implementáciou som navrhol celkovú architektúru systému, ktorá definuje tok dát od GPS trackerov až po webovú aplikáciu.

\subsection{Prehľad komponentov}
\label{sec:PrehladKomponent}

Systém pozostáva zo šiestich hlavných vrstiev:

\begin{enumerate}
    \item \textbf{End Devices} -- SenseCap T1000-B GPS trackery zbierajúce pozičné dáta
    \item \textbf{LoRaWAN Gateway} -- RAK7289 V2 most medzi rádiovými signálmi a~IP sieťou
    \item \textbf{MQTT Broker} -- mqtt.hedurio.com sprostredkúva správy medzi gateway a~backendom
    \item \textbf{Backend Server} -- Python/Flask aplikácia spracujúca dáta a~poskytujúca API
    \item \textbf{Databáza} -- PostgreSQL s~rozšíreniami TimescaleDB a~PostGIS
    \item \textbf{Frontend} -- React webová aplikácia pre vizualizáciu a~správu
\end{enumerate}

\subsection{Dátový tok}
\label{sec:DatovyTok}

Životný cyklus GPS správy prechádza nasledujúcimi fázami: tracker získa GPS fix a~zostaví payload, payload je odoslaný cez LoRaWAN na frekvenčnom pásme EU868, gateway prijme rádiovú správu a~publikuje dekódovaný JSON na MQTT topic, backend prijme správu a~spustí parser, dáta sú uložené do PostgreSQL a~WebSocket broadcastuje novú pozíciu všetkým pripojeným klientom. Táto architektúra zabezpečuje latenciu pod 2~sekundy od získania GPS fixu po zobrazenie na mape.

\section{Hardware Setup a~Konfigurácia}
\label{sec:HardwareSetup}

Prvou fázou projektu bola konfigurácia hardvérovej infraštruktúry. Táto fáza vyžadovala pochopenie LoRaWAN protokolu a~riešenie niekoľkých neočakávaných problémov.

\subsection{LoRaWAN protokol}
\label{sec:UvodLoRaWAN}

LoRaWAN (Long Range Wide Area Network) je bezdrôtový komunikačný protokol navrhnutý pre IoT zariadenia s~nízkym energetickým odberom~\cite{lorawan_spec}. Na rozdiel od WiFi či Bluetooth, ktoré majú dosah desiatky metrov, LoRaWAN umožňuje komunikáciu na vzdialenosti až 15~kilometrov v~otvorenom teréne.

Pre tento projekt som zvolil LoRaWAN z~niekoľkých dôvodov. Prvým bol dlhý dosah -- až 15~km v~Line of Sight podmienkach, 2-5~km v~mestskom prostredí. Druhým bola nízka spotreba energie -- trackery vydržia na batérii dni až týždne. Tretím dôvodom bolo nelicencované pásmo EU868, ktoré nevyžaduje licenciu pre prevádzku v~Európe. Na celodenných eventoch nie je možné priebežné nabíjanie zariadení, takže energetická efektívnosť bola prioritou.

\subsection{Gateway RAK7289 V2}
\label{sec:Gateway}

Firma mi poskytla gateway RAK7289 V2 vo variante C2 s~integrovaným LTE modulom. Varianta C2 je hardvérová konfigurácia so 4G konektivitou, čo eliminuje potrebu ethernet pripojenia a~umožňuje nasadenie kdekoľvek s~mobilným signálom.

\subsection{Problémy a~riešenia}
\label{sec:HWProblemy}

Počas konfigurácie hardvéru som narazil na niekoľko problémov.

\textbf{GPS cold start:} Pri prvom zapnutí tracker posielal súradnice (0.0, 0.0), kým GPS modul nezískal fix. Toto je bežné správanie pri cold start -- GPS modul nemá uložené údaje o~satelitoch a~potrebuje čas na ich vyhľadanie. Riešením bolo filtrovanie nulových súradníc v~parseri.

\textbf{Firmware bug batérie:} Podľa diskusií na SenseCraft fóre niektoré trackery hlásili 100\% batérie aj keď boli takmer vybité. Implementoval som detekciu tejto anomálie -- ak tracker hlási 100\% batériu viac ako 24 hodín bez zmeny, systém ho označí ako podozrivý.

\textbf{GPS presnosť v~budovách:} Presnosť GPS vnútri budov dosahovala len približne 300 metrov, čo mentor potvrdil: \emph{``GPS v~budově tady u~toho spíš nepůjde. A~když náhodou ano tak přesnost bude tak 300~m.''}

\section{MQTT a~Parser}
\label{sec:MQTTParser}

Po úspešnej konfigurácii hardvéru som pristúpil k~implementácii softvérovej časti pre príjem a~spracovanie dát.

\subsection{MQTT Protokol}
\label{sec:MQTTProtokol}

MQTT (Message Queuing Telemetry Transport) je ľahký publish/subscribe protokol ideálny pre IoT aplikácie~\cite{mqtt_spec}. Na rozdiel od REST API, kde klient aktívne dopytuje server (polling), MQTT umožňuje serveru posielať dáta klientom okamžite po ich prijatí (push).

Výhody MQTT pre tento projekt zahŕňali nízku latenciu s~doručením správ v~milisekundách, efektivitu s~minimálnou hlavičkou protokolu iba 2 bajty, spoľahlivosť vďaka QoS úrovniam garantujúcim doručenie a~TLS podporu pre šifrovanú komunikáciu.

\subsection{Pripojenie na broker}
\label{sec:BrokerPripojenie}

Pre pripojenie na firemný broker mqtt.hedurio.com som použil Python knižnicu paho-mqtt s~TLS šifrovaním:

\begin{lstlisting}[language=Python, caption={Inicializácia MQTT klienta s~TLS}]
import paho.mqtt.client as mqtt
import ssl

class MQTTClient:
    def __init__(self, config):
        self.client = mqtt.Client(client_id=config.CLIENT_ID)

        # TLS konfiguracia s certifikatmi
        self.client.tls_set(
            ca_certs=config.CA_CERT_PATH,
            certfile=config.CLIENT_CERT_PATH,
            keyfile=config.CLIENT_KEY_PATH,
            tls_version=ssl.PROTOCOL_TLS
        )

        # Autentifikacia
        self.client.username_pw_set(
            config.MQTT_USERNAME, config.MQTT_PASSWORD)

        # Callback funkcie
        self.client.on_connect = self._on_connect
        self.client.on_message = self._on_message
\end{lstlisting}

Po úspešnom pripojení sa klient prihlási na odber topicu application/SENSECAP/decoded, kde gateway publikuje dekódované správy z~trackerov vo formáte JSON.

\subsection{Parser SenseCap Payloadov}
\label{sec:Parser}

Správy z~trackerov prichádzajú vo formáte JSON s~vnorenou štruktúrou. GPS súradnice a~telemetria sú uložené v~poli messages s~unikátnymi measurement ID. Latitude má ID 4198, longitude 4197 a~battery level 3000.

\begin{lstlisting}[language=Python, caption={Validácia GPS súradníc s~ošetrením edge cases}]
def validate_coordinates(lat: float, lon: float) -> bool:
    """Validuje GPS suradnice - filtruje cold start a garbage"""
    # Cold start - tracker este nema GPS fix
    if lat == 0.0 and lon == 0.0:
        return False
    # Garbage values z firmware
    if lat == 999.0 or lon == 999.0:
        return False
    # Validny rozsah WGS84
    return -90 <= lat <= 90 and -180 <= lon <= 180
\end{lstlisting}

\section{Databáza}
\label{sec:Databaze}

Návrh databázovej vrstvy bol kritickou časťou projektu. Potreboval som efektívne ukladať časové rady GPS pozícií, podporovať geospatial operácie a~škálovať pre potenciálne tisíce trackerov.

\subsection{Výber technológií}
\label{sec:VyberTechnologii}

Po konzultácii s~mentorom som zvolil PostgreSQL 16~\cite{postgresql_docs} s~rozšíreniami TimescaleDB a~PostGIS. Mentor potvrdil správnosť voľby: \emph{``PostGIS a~TimescaleDB v~praktických záležitostech používáme celkem dost a~ušetří ti to strašně moc práce.''}

\textbf{Prečo TimescaleDB:} TimescaleDB~\cite{timescaledb_docs} je rozšírenie PostgreSQL optimalizované pre časové rady. GPS pozície sú klasickým príkladom -- každý záznam má timestamp a~postupne pribúdajú nové dáta. Výhodami sú hypertables s~automatickým rozdelením dát do chunks podľa času, kompresia až 95\% úspora miesta pre staršie dáta a~retention policies pre automatické mazanie starých dát.

\textbf{Prečo PostGIS:} PostGIS~\cite{postgis_docs} rozširuje PostgreSQL o~geospatial funkcie. Namiesto ukladania súradníc ako FLOAT poskytuje natívny dátový typ GEOGRAPHY, ktorý umožňuje efektívne priestorové dotazy ako ST\_Distance pre výpočet vzdialenosti a~ST\_Contains pre detekciu či bod leží v~polygóne.

\subsection{Databázová schéma}
\label{sec:Schema}

Schéma databázy obsahuje 8 tabuliek. Hlavné sú devices pre zoznam trackerov a~ich metadáta, positions ako TimescaleDB hypertable pre GPS záznamy, geofences pre virtuálne zóny uložené ako PostGIS polygóny a~poi pre body záujmu.

\begin{lstlisting}[language=SQL, caption={Vytvorenie TimescaleDB hypertable pre GPS pozície}]
-- Vytvorenie zakladnej tabulky
CREATE TABLE positions (
    id BIGSERIAL,
    device_id INTEGER NOT NULL REFERENCES devices(id),
    latitude DOUBLE PRECISION NOT NULL,
    longitude DOUBLE PRECISION NOT NULL,
    location GEOGRAPHY(POINT, 4326),
    battery INTEGER,
    timestamp TIMESTAMPTZ NOT NULL,
    PRIMARY KEY (id, timestamp)
);

-- Konverzia na hypertable s tyzdnovymi chunks
SELECT create_hypertable('positions', 'timestamp',
    chunk_time_interval => INTERVAL '1 week');

-- Automaticka kompresia dat starsich ako 7 dni
SELECT add_compression_policy('positions', INTERVAL '7 days');
\end{lstlisting}

\subsection{PostGIS Trigger}
\label{sec:PostGISTrigger}

Pre automatické vytvorenie PostGIS POINT z~lat/lon súradníc som implementoval trigger:

\begin{lstlisting}[language=SQL, caption={PostGIS trigger pre automatické vytvorenie GEOGRAPHY}]
CREATE OR REPLACE FUNCTION populate_location()
RETURNS TRIGGER AS $$
BEGIN
    NEW.location := ST_SetSRID(
        ST_MakePoint(NEW.longitude, NEW.latitude), 4326
    )::geography;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_populate_location
    BEFORE INSERT ON positions
    FOR EACH ROW EXECUTE FUNCTION populate_location();
\end{lstlisting}

def broadcast_new_position(position_data: dict):
    """Broadcast novej pozicie vsetkym klientom"""
    socketio.emit('new_position', {
        'device_id': position_data['device_id'],
        'latitude': position_data['latitude'],
        'longitude': position_data['longitude'],
        'battery': position_data['battery'],
        'timestamp': position_data['timestamp'].isoformat()

    # Notifikacia pre geofence eventy
    if position_data.get('geofence_event'):
        socketio.emit('geofence_alert', {
            'type': position_data['geofence_event']['type'],
            'zone_name': position_data['geofence_event']['zone_name']
        })

\section{Geofencing a~POI}
\label{sec:GeofencingPOI}

Geofencing je jednou z~najhodnotnejších funkcionalít systému pre praktické nasadenie na eventoch.

\subsection{Koncept Geofencing}
\label{sec:GeofencingKoncept}

Geofencing je technika vytvárania virtuálnych geografických zón a~automatickej detekcie, keď zariadenie do zóny vstúpi alebo ju opustí. Pre eventy má široké využitie: detekcia vstupu do backstage area len pre authorized personál, monitorovanie pohybu security personálu, štatistiky návštevnosti jednotlivých zón a~alerting pri opustení definovaného perimetra.

\label{sec:GeofencingImpl}

Pre kreslenie zón som implementoval intuitívne rozhranie. Proces vytvorenia zóny zahŕňa kliknutie na Draw New Zone, pridávanie bodov polygónu kliknutím na mapu, uzavretie polygónu dvojklikom a~vyplnenie formulára pre názov, farbu a~popis zóny.

\subsection{Bezpečnostné opatrenia}
\label{sec:Bezpecnost}

Implementoval som niekoľko bezpečnostných opatrení: TLS pre šifrovanú MQTT komunikáciu na porte 8883, parameterized queries pre ochranu pred SQL injection, CORS s~povolenými len konkrétnymi origins, environment variables bez hardcoded credentials a~input validation pre validáciu všetkých vstupov z~API.

\section{Testovanie}
\label{sec:Testovani}

Testovanie prebiehalo kontinuálne počas celého vývoja -- od unit testov parsera po real-world testy s~fyzickými trackermi.

\subsection{GPS Simulátor}
\label{sec:Simulator}

Pre vývoj bez fyzických trackerov som vytvoril GPS simulátor, ktorý generuje realistické testovacie dáta. Simulátor umožňuje definovať východziu pozíciu, rýchlosť pohybu a~generuje náhodné trajektórie. Toto výrazne urýchlilo vývoj, pretože som nemusel čakať na reálne payloady z~trackerov.

\subsection{Real-world Testing}
\label{sec:RealTesting}

Okrem simulátora som systém testoval s~reálnymi trackermi v~teréne. Testy odhalili, že presnosť GPS v~budovách je výrazne horšia až 300m, batéria vydrží približne 2-3 dni pri 5-minútovom intervale, LoRaWAN dosah je v~mestskom prostredí cca 2-3 km a~cold start GPS trvá 30-120 sekúnd.

Mentor potvrdil reálne parametre dosahu: \emph{``To můžu potvrdit, mám 10 km v~LoS.''}
